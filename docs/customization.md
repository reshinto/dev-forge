# Customization

claude-forge is designed to be adapted. Project-level files always take precedence over plugin-level files. You can override individual skills, extend rules, disable specific hooks, and layer additional plugins alongside claude-forge — all without modifying the plugin source.

> **Prerequisites**
> - claude-forge installed and scaffold complete (see [Getting Started](getting-started.md))

---

## Override Model: Project Files Win

When Claude Code loads configuration, it merges files in this order (later entries win):

1. Plugin defaults (inside the `claude-forge` plugin directory)
2. Project `.claude/` files (in your repository)
3. Local overrides (`.claude/settings.local.json`, gitignored)

Any file you create under `.claude/` in your project overrides the plugin's equivalent. You never need to edit the plugin source.

---

## Overriding Skills

To replace a skill with a project-specific version, create a `SKILL.md` at the same name inside your project's `.claude/` skills directory:

```
.claude/skills/tdd/SKILL.md
```

Claude Code will load your version instead of the plugin's version. The file format is the same — a YAML frontmatter block followed by the skill instructions:

```markdown
---
name: tdd
description: Project-specific TDD workflow
user-invocable: true
---

[Your custom instructions here]
```

To extend rather than replace, copy the plugin's `SKILL.md` into your project and append your additions.

---

## Adding Project-Specific Rules

Drop any Markdown file into `.claude/rules/` and Claude Code picks it up automatically:

```
.claude/rules/my-domain-rules.md
```

Rules files are loaded as context at session start. Use them to encode project conventions that are too specific for the plugin's generic templates — for example, naming conventions for a domain model, required code comment patterns, or API usage constraints.

Reference the rule file from `CLAUDE.md` if you want Claude to be explicitly aware of it:

```markdown
## Rules

See `.claude/rules/my-domain-rules.md` for domain-specific constraints.
```

## Adjusting Hook Behavior

**To disable a specific hook**, remove or comment out its entry in `.claude/settings.json`. Hook entries follow this shape:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          { "type": "command", "command": "bash ...", "timeout": 10 }
        ]
      }
    ]
  }
}
```

Delete the hook object you want to disable. Your `settings.json` is project-owned — changes here do not affect the plugin source.

**To add a new hook**, append a new entry to the appropriate lifecycle array (`SessionStart`, `PreToolUse`, `PostToolUse`) in `.claude/settings.json`. The hook script can live anywhere — conventionally in `.claude/hooks/`:

```json
{ "type": "command", "command": "bash .claude/hooks/my-custom-check.sh", "timeout": 10 }
```

**To adjust the unified gate sequence**, edit `.claude/hooks/session-end-unified-gate.sh` directly. This file is generated by `init.sh` and is project-owned. Add, remove, or reorder steps as needed.

## Adding Plugins Alongside claude-forge

Multiple plugins can be active in the same session. Register additional plugins in `.claude/settings.json` under the `plugins` key:

```json
{
  "plugins": ["claude-forge", "my-other-plugin"]
}
```

Skills and agents from all registered plugins are available simultaneously. If two plugins define a skill with the same name, the plugin listed last wins. To resolve conflicts explicitly, create a project-level override (see "Overriding Skills" above).

Plugin profiles in `.claude/hooks/plugin-profiles.json` can also include additional plugins per branch pattern:

```json
{
  "profiles": [
    { "pattern": "feat/", "plugins": ["claude-forge", "my-other-plugin"] }
  ]
}
```

## CLAUDE.md Guidance

`CLAUDE.md` is the primary instruction file Claude reads at session start. The scaffold generates a starter version, but you should treat it as a living document owned by your team.

Key sections to maintain:

| Section | What to put there |
|---|---|
| Project identity | Name, one-line description, tech stack |
| General behavior | Any "always do X" or "never do Y" instructions |
| Key paths | Source directories, registry files, important config |
| Rules | References to files under `.claude/rules/` |
| Architecture notes | Patterns Claude should know before writing code |

Keep CLAUDE.md under 150 lines. Move detailed constraints into `rules/` files and link to them. Long CLAUDE.md files dilute attention — shorter is more reliable.
